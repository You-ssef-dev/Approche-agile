📌 Section 3

🎯 3 Axes Stratégiques du DevOps

Intégration continue, déploiement continu et Infrastructure as Code.

🔄 CI — Intégration Continue

  • Changements poussés fréquemment
  • Tests automatisés exécutés
  • Échec → livraison rejetée
  • Succès → code intégré

Types de tests : Fonctionnel, Technique, Qualité, Sécurité, Performance, UX

🚀 CD — Déploiement Continu

  • Déploiement automatisé
  • Infrastructure & sécurité gérées
  • Backup & rollback supportés
  • Cache & performance optimisés

🏗️ IaC — Infrastructure as Code

  • Infrastructure scriptée
  • Versionnée & reproductible
  • Auto-scaling automatique
  • Environnements reproductibles

📌 Section 6

⚡ DevOps vs Agile

Le DevOps étend l'Agile en incluant les opérations.

Critère Agile DevOps
Focus Développement Développement + Opérations
Collaboration Client & développeurs Dev, Ops, QA, Sécurité
Méthode Sprints CI/CD automatisé
Automatisation Limitée Complète
Infrastructure Non concerné Infrastructure as Code
Déploiement Manuel Automatisé & fiable
💡 DevOps étend Agile : l'Agile améliore le développement, le DevOps automatise le chemin jusqu'à la production.
📌 Section 7

🔀 CI vs Continuous Delivery vs Continuous Deployment

Trois niveaux d'automatisation dans la chaîne DevOps.

🔄 Continuous Integration

  • Code poussé au repository
  • Build + tests automatiques
  • Détection précoce des bugs

📦 Continuous Delivery

  • Code préparé pour la production
  • Approbation manuelle requise
  • Release contrôlée

🚀 Continuous Deployment

  • Déploiement 100% automatisé
  • Aucune approbation manuelle
  • En production immédiatement

📘 Chapitre 2 — GitLab CI/CD

Pipelines, stages, jobs, runners et configuration

📌 Section 8

🔀 GitLab CI/CD

GitLab automatise le build, les tests et le déploiement via des pipelines.

🔀 GitLab CI/CD

  • Automatise build, test, deploy
  • Utilise des pipelines
  • Exécuté par des runners
  • Configuré via .gitlab-ci.yml

📋 Prérequis

  • Un compte sur GitLab.com
  • Un repository Git (projet)
  • Fichier .gitlab-ci.yml à la racine
  • Un runner configuré
📌 Section 9

🔗 Architecture d'un Pipeline

Un pipeline est une séquence d'étapes automatisées, déclenchée à chaque push.

🔗 Flux du Pipeline

🔨 Build
→
🧪 Test
→
🚀 Deploy
→
✅ Validate
1

Pipeline

Processus de haut niveau

2

Stages

Groupes de jobs (séquentiels)

3

Jobs

Tâches exécutées dans les stages

⚙️ Runner

Application qui exécute les jobs du pipeline (peut être local, VM, container).

📄 Artifacts

Fichiers générés par les jobs (rapports de couverture, logs, résultats de tests).

📌 Section 10

📝 Fichier .gitlab-ci.yml

Configuration du pipeline CI/CD, placé à la racine du projet.

📝 Exemple de base

# Définition des étapes stages: - build - test # Job de build build-job: stage: build script: - echo "Compiling the code..." - echo "Compile complete." # Job de tests unit-test-job: stage: test script: - vendor/bin/phpunit tests - echo "tests terminés"
📌 GitLab détecte automatiquement le fichier .gitlab-ci.yml à la racine et exécute le pipeline à chaque commit.
📌 Section 11

⚙️ GitLab Runner

Programme qui exécute les jobs du pipeline CI/CD.

⚙️ Qu'est-ce qu'un Runner ?

  • Composant qui exécute les tâches CI/CD
  • Développé en Go
  • Installable sur serveur, VM, container
  • Reste en écoute du GitLab pour les tâches
  • Peut être partagé ou spécifique

🖥️ Installation Windows

# 1. Télécharger gitlab-runner.exe # 2. Placer dans C:/Gitlab-Runner # 3. En tant qu'Administrateur : .\gitlab-runner.exe install .\gitlab-runner.exe start .\gitlab-runner.exe register

▶️ Démarrer le Pipeline

# Lancer le runner gitlab-runner.exe run # Dans GitLab : CI/CD → Pipelines → "Run pipeline"

L'état d'exécution est visible dans le menu CI/CD → Pipelines de GitLab.

🔥 Pipeline Avancé

Pipeline avec couverture de code et intégration SonarQube.
